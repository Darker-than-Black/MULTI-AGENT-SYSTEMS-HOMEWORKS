## Summary
This code review evaluates the implementation of a Research Agent project based on the requirements specified in the `README.md` file. The focus is on the correctness of the server applications and agent systems, as well as adherence to the outlined tasks.

## Key Findings

### 1. **README.md Requirements**
The `README.md` specifies several key tasks for the project:
- **Custom ReAct Loop**: The project replaces `create_react_agent` with a custom implementation, which is evident in the `main.ts` file where the `runAgentTurn` function is called to manage agent interactions.
- **Tools as JSON Schema**: The implementation appears to define tools in a structured manner, although specific JSON Schema definitions were not directly visible in the provided files.
- **Dialogue Memory Management**: The `createSessionMemory` function is utilized to manage dialogue context, which aligns with the requirement to handle memory without `MemorySaver`.
- **System Prompt**: The project includes a mechanism to improve the system prompt, but the specifics of the prompt's effectiveness were not evaluated in the provided files.
- **Logging**: The implementation logs tool calls and responses, which is evident in the `main.ts` file where console logs are used to track agent responses and report saving.
- **Error Handling**: The code includes error handling for both application errors and report saving, ensuring that tool errors do not crash the agent.

### 2. **Server Application Structure**
- The `main.ts` file serves as the entry point for the application, managing user input and agent interactions effectively.
- The use of `async/await` for handling asynchronous operations is appropriate, ensuring that the application remains responsive.
- The structure of the project, with separate directories for agents, tools, and utilities, promotes modularity and maintainability.

### 3. **Agent System Implementation**
- The agent system appears to follow a clear flow, processing user input, invoking the agent's logic, and managing memory effectively.
- The `runAgentTurn` function is central to the agent's operation, although the specifics of its implementation were not reviewed in detail as the file was not fetched.

### 4. **Architecture Checks**
- The scripts `check-architecture-invariants.sh` and `check-architecture-sync.sh` are designed to enforce architectural constraints, ensuring that the code adheres to the specified design principles.
- The invariant checks ensure that loops are only present in designated files and that tools do not directly import LLM dependencies, which is a good practice for maintaining separation of concerns.

## Limitations
- The review did not include a detailed examination of all source files, particularly those in the `src/agent` directory, which may contain critical logic for the agent's behavior.
- The effectiveness of the system prompt and the JSON Schema definitions for tools were not thoroughly evaluated due to the lack of direct visibility into those implementations.
