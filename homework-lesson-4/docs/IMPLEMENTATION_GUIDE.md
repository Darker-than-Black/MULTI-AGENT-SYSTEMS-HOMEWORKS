# План реалізації (блоками з checkpoint’ами)
## 0: Базова підготовка і “контракт” архітектури
* Фіксуємо цільову мінімальну архітектуру на базі поточних файлів у src/agent, src/tools, src/config, src/utils.
* Узгоджуємо формат messages, tool_calls, tool_result, щоб один формат використовувався всюди.
* Прибираємо зайві моки, які конфліктують з майбутнім real flow.
* Перевірка: npm run check проходить, npm run dev стартує без падіння.
* Доопрацювання при проблемах: вирівнюємо типи в types.ts, поки всі модулі не компілюються стабільно.

## 1: Реальні tools + JSON Schema
* Реалізуємо web_search, read_url, write_report із реальним виконанням (не моки).
* Узгоджуємо JSON Schema в schemas.ts з фактичними аргументами функцій.
* Додаємо базову валідацію аргументів і дружні помилки tool-рівня.
* Перевірка: ручний виклик кожного tool із main.ts/тимчасового smoke сценарію, write_report реально створює файл в output/.
* Доопрацювання: якщо schema не збігається з runtime-аргументами, правимо schema першою, потім dispatcher.

## 2: LLM client і tool calling через API
* Реалізуємо llm-client.ts під обраний провайдер із передачею messages + tools(JSON Schema).
* Нормалізуємо відповідь провайдера в внутрішній LlmTurnResult (assistant text + tool calls).
* Додаємо обробку API-помилок (rate limit, network, invalid response).
* Перевірка: окремий smoke-кейс “один запит -> один tool call” і “один запит -> одразу final answer”.
* Доопрацювання: якщо парсинг нестабільний, додаємо адаптер-слой між raw API і LlmTurnResult.

## 3: Власний ReAct loop (ядро ДЗ)
* Допрацьовуємо run-agent.ts: цикл LLM -> tool calls -> tool results -> LLM до фіналу.
* Додаємо MAX_ITERATIONS, stop conditions, і graceful finish при ліміті.
* Гарантуємо, що tool result повертається у messages у правильному форматі ролі tool.
* Перевірка: сценарій з 3-5+ викликами tools на одному запиті реально проходить до фінальної відповіді.
* Доопрацювання: якщо агент зациклюється, посилюємо stop rules і prompt-обмеження.

## 4: Пам’ять діалогу + інтерактивний CLI
* Реалізуємо сесійне збереження messages між запитами в memory.ts.
* Робимо повноцінний REPL у main.ts: багато user turns, exit/quit, показ відповіді агента.
* Додаємо fallback: якщо агент не викликав write_report, можна зберегти звіт із CLI.
* Перевірка: 2-3 послідовні запити, де другий явно залежить від контексту першого.
* Доопрацювання: якщо контекст росте занадто швидко, додаємо обрізання/компресію історії.

## 5: Prompt engineering (покращений system prompt)
* Переписуємо prompt.ts: роль, правила, формат дій, обмеження, критерії “коли викликати tools”.
* Додаємо короткі приклади очікуваної поведінки (few-shot style, компактно).
* Підлаштовуємо prompt під вимогу: факти через tools, структурований markdown-звіт, список джерел.
* Перевірка: порівнюємо якість відповіді “до/після prompt” на 2 однакових запитах.
* Доопрацювання: якщо агент робить мало tool calls або фантазує, посилюємо правила evidence-first.

## 6: Логування, стабільність, фіналізація ДЗ
* Доробляємо logger.ts: видно tool name, args, result/error, номер ітерації.
* Закриваємо edge cases: invalid tool args, failed URL read, empty search result, API timeout.
* Оновлюємо README під TS запуск (npm run dev), додаємо приклад консолі та приклад output-звіту.
* Перевірка: end-to-end demo сценарій з генерацією файлу звіту в output/ без падіння застосунку.
* Доопрацювання: точково правимо блок, який не проходить demo, і повторюємо E2E до стабільного результату.
