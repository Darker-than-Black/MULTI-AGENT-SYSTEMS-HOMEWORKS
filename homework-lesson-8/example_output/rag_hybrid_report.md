## Summary
RAG (Retrieval-Augmented Generation) — це підхід, де LLM не лише генерує текст, а й спочатку звертається до зовнішніх документів/індексів (retrieval), щоб підсилити відповідь актуальними та релевантними фактами.

## Key Findings
### 1) Базове визначення та pipeline (з локальної knowledge base)
З локальних матеріалів:
- **RAG** визначається як техніка, що дозволяє LLM **отримувати та включати нову інформацію з зовнішніх джерел** через механізм retrieval.
- **Процес**: (а) викликається **retriever**, який обирає релевантні документи під запит користувача; (б) ці документи використовуються для **доповнення (augmentation) промпта**; (в) LLM генерує відповідь на основі цього контексту.
- Для retrieval можуть застосовуватись **різні методи**, що залежать від типу індекса.

### 2) Основні підходи до retrieval
На практиці retrieval у RAG зазвичай будується як комбінація кількох стратегій:
- **Sparse / lexical retrieval** (на кшталт BM25): добре знаходить точні збіги термінів/назв.
- **Dense / semantic retrieval** (векторні ембедінги): краще знаходить документи, що відповідають за змістом, навіть якщо формулювання відрізняються.
- **Hybrid retrieval**: поєднує sparse + dense сигнали, щоб закрити різні типи невідповідностей (лексичні та семантичні).
- Додатково часто застосовують **reranking**: після первинного retrieval (щоб не рахувати дорого на всьому корпусі) модель переупорядковує топ-кандидати за релевантністю.

### 3) Чим hybrid search кращий за чистий semantic search
З веб-джерел (сучасні RAG-підходи):
- У TREC RAG 2025 учасники порівнювали **BM25**, **dense (DPR embeddings)** та **hybrid (sparse+dense + cross-encoder reranking)**; у звіті зазначено, що **hybrid retrieval дав найкращі результати** (MAP 0.1037, nDCG@30 0.527, Recall@100 0.158) та підкреслено важливість **комбінації лексичних і семантичних сигналів**.
- У практичному огляді про RAG у продакшені описано, що **most RAG failures are retrieval failures**, а **hybrid** з BM25 + dense та ф’южном результатів (на кшталт Reciprocal Rank Fusion) є “найбільш відчутним” покращенням над naive RAG. Логіка: dense краще для семантичного відповідника, sparse — для точних ключових слів/термінів/версій/імен.
- У LiveRAG Challenge 2025 (опис підходу) також використано **hybrid retrieval (BM25 + E5)** як фінальну стратегію з огляду на компроміс “якість vs обчислювальна вартість”: neural re-ranking дав сильний приріст, але був надто дорогим, тому обрали hybrid як більш практичний baseline.

**Висновок:** hybrid search зазвичай кращий за чистий semantic search, бо **комплементарно** покриває:
- випадки, коли користувач формулює запит інакше (dense допомагає),
- випадки, коли потрібні точні терміни/імена/числа/версії (sparse допомагає),
- і дає більш надійний набір кандидатів для подальшого reranking/генерації.

## Порівняння локальних матеріалів з веб-джерелами
- **Локальна база** дає загальну рамку RAG: retriever → augmentation промпта → генерація, і згадує, що retrieval може бути різним за методами.
- **Веб-джерела** деталізують сучасну інженерію retrieval у RAG: чому саме hybrid (BM25+dense) та як часто додають reranking; також наводять приклади з актуальних змагань/публікацій (TREC RAG 2025, LiveRAG Challenge 2025) з порівняннями sparse/dense/hybrid.
