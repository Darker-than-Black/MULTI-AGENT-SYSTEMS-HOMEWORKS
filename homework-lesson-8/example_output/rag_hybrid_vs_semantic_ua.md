## Summary
RAG (Retrieval-Augmented Generation) — це підхід, де LLM генерує відповідь, спираючись на зовнішньо знайдені фрагменти знань (retrieved context), отримані через retrieval-модуль. Якість RAG критично залежить від того, наскільки добре retriever знаходить релевантні документи/чанки.

## Key Findings
### 1) Визначення RAG і типова pipeline (локальні матеріали)
- RAG підсилює LLM механізмом інформаційного пошуку: модель отримує релевантні зовнішні документи/чанки і використовує їх у промпті для генерації відповіді.
- Типова pipeline включає: **chunking** (розбиття контенту на чанки), **embeddings** (векторизація чанків), **retrieval** (пошук релевантних чанків за запитом), **reranking** (перевпорядкування кандидатів за релевантністю) та **generation** (LLM-генерація з використанням знайденого контексту).
- Локальні матеріали прямо вказують, що **чистий vector search може пропускати ключові факти**, і це можна компенсувати **традиційним text search**, додаючи результати до чанків, пов’язаних із retrieved vectors.

### 2) Основні підходи до retrieval
Зібрані локальні та web-дані узгоджуються з такою картиною:
- **Лексичний / keyword retrieval (BM25, full-text)**: добре працює для точних збігів термінів, назв, дат, спеціалізованого жаргону.
- **Dense / semantic retrieval (vector search, KNN)**: знаходить семантично близькі фрагменти навіть коли точних ключових слів немає.
- **Sparse encoder / learned sparse retrieval**: семантика кодується у розрідженому (sparse) представленні, що поєднує переваги індексації та семантичного узгодження.
- **Hybrid retrieval**: комбінує keyword/full-text і vector/semantic пошук.
- **Reranking (друга стадія)**: після швидкого retrieval (candidate generation) застосовується reranker, який оцінює релевантність кандидатів і покращує порядок контексту для LLM.
- **Query rewriting / iterative/agentic retrieval**: у сучасних системах запит може декомпозуватися/переформульовуватися для підвищення якості пошуку (згадується як частина сучасних RAG-підходів у web-оглядах).

### 3) Порівняння: hybrid vs pure semantic search
**Чому hybrid зазвичай кращий за чистий semantic/vector search** (з доказами з web-джерел):
- **Hybrid поєднує сильні сторони обох підходів**:
  - vector search дає семантичну близькість навіть без keyword-матчів;
  - keyword/full-text дає точність для випадків, де важливі точні збіги (коди, імена, дати, вузькі терміни).
- **Технічно hybrid виконує full-text і vector запити паралельно та зливає результати**. У Azure AI Search це робиться через **Reciprocal Rank Fusion (RRF)**.
- **Reranking підсилює ефект hybrid**: навіть якщо candidate set зібраний з різних retrievers, reranker допомагає відсортувати найрелевантніші чанки перед тим, як їх використає LLM.
- **Локальні матеріали підтверджують практичну проблему**: vector database search може пропускати ключові факти; додавання traditional text search зменшує цей ризик.

### 4) Підсумок порівняння локальних vs web-джерел
- **Узгодження**:
  - і локальні матеріали, і web-джерела підкреслюють, що retrieval є критичним для якості RAG;
  - обидва підходи підтримують ідею: **hybrid (keyword + semantic) зменшує пропуски** та покращує релевантність контексту;
  - reranking розглядається як важливий етап для підвищення точності.
- **Відмінності**:
  - локальні матеріали фокусуються на практичній інженерній мотивації (vector search може пропускати факти) та загальній структурі pipeline;
  - web-джерела дають більше деталей про конкретні механізми (напр., RRF у hybrid search) та сучасні методи (напр., Blended RAG як комбінація dense/sparse з hybrid query стратегіями).
